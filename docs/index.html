Sample interview questions — click any question to expand the answer

1. What's the difference between Opus, Sonnet, and Haiku — and how do you choose?
Opus is the most capable — pick it for complex reasoning, long-horizon agentic work, and deep coding tasks. Sonnet balances intelligence and cost — the default for most production workloads and agents. Haiku is fastest and cheapest — ideal for classification, routing, and sub-agent roles. Strategy: start with Sonnet, escalate to Opus where evals show quality gaps, downgrade to Haiku where latency or cost dominates.
2. How does prompt caching work and when should you use it?
Mark stable prefix content — system instructions, large documents, tool definitions — with a cache_control breakpoint. Anthropic stores that prefix server-side and serves later requests with the same prefix at reduced input token cost and lower latency. Use it whenever you have repeated large context: RAG systems, agentic loops with stable tool schemas, and multi-turn chats with fixed system prompts. Cache TTL is 5 minutes by default, refreshed on each hit.
3. Why does Anthropic recommend XML tags in prompts?
Claude is trained to treat XML as semantic structure. Wrapping content in tags like , , clearly separates context, examples, task, and output requirements. This reduces ambiguity, improves instruction following, and makes outputs easier to parse programmatically when you ask Claude to respond inside specific tags.
4. How do you reduce hallucinations in Claude's outputs?
Compound several techniques: (1) provide source material in the prompt and instruct Claude to cite it, (2) give Claude explicit permission to say "I don't know," (3) ask it to reason step-by-step before answering, (4) enable extended thinking for complex reasoning, (5) ground claims via retrieval, (6) run judge-model evals and tune. The strongest intervention is usually giving Claude the information it needs rather than relying on parametric memory.
5. What's the difference between system messages and user messages?
The system parameter sets persistent behavior, role, constraints, and tone — outside the messages array. The messages array holds alternating user/assistant turns. Best practice: stable instructions, persona, output format, and policies go in system; actual tasks, data, and conversation history go in messages. System content also caches well.
6. What is CLAUDE.md and why does it matter?
CLAUDE.md is a project memory file Claude Code reads automatically at session start. Put your project's architecture, coding conventions, key directories, important commands, and team rules inside it. Every new conversation begins with this context loaded — eliminating repetitive setup. A user-level file lives at ~/.claude/CLAUDE.md; project-level files override user-level.
7. How do you optimize token usage in long Claude Code sessions?
Use /compact to summarize prior context instead of /clear when you want to keep a thread. Write precise prompts targeting specific files. Move recurring instructions into CLAUDE.md. Limit simultaneously enabled MCP servers — too many tool definitions can shrink usable context from 200K to under 100K tokens. Use sub-agents for isolated explorations that would otherwise pollute the main session.
8. What are hooks in Claude Code and what are they used for?
Hooks are shell commands triggered by Claude Code lifecycle events — before/after a tool call, before a commit, on session start. Use them to enforce linting, run tests, log actions for audits, block dangerous commands, or notify CI systems. They're defined in settings.json and let you wrap Claude's autonomy in deterministic guardrails.
9. What is Model Context Protocol (MCP)?
MCP is an open standard for connecting AI agents to external tools and data sources. An MCP server exposes tools, resources, and prompts; a client (Claude Code, the API's MCP connector, or the Claude.ai interface) discovers and calls them. Transports include stdio (local processes), SSE, and HTTP. It eliminates one-off integrations — any MCP-compliant server works with any MCP-compliant client.
10. How would you secure an MCP server in production?
Treat MCP servers like any privileged subprocess: install only from trusted sources, use scoped credentials with minimum required permissions, store secrets in env vars or a secrets manager (not plaintext configs), enable structured logging for all tool calls, restrict network egress, and run processes in isolated containers for high-risk integrations. Pin server versions to avoid supply-chain surprises.
11. How would you design an agentic system with Claude?
Start with explicit success criteria. Build the tool surface (typed JSON schemas, descriptive names, narrow scope per tool). Pick a model — usually Sonnet for the orchestrator, Haiku for cheap sub-tasks. Implement the loop: prompt → model → tool call → execute → feed result back → repeat until done or max iterations. Add guardrails: step cap, cost ceiling, timeout, observability. Write evals before you scale. Parallelize with sub-agents.
12. What are sub-agents and when should you use them?
Sub-agents are isolated Claude instances with their own context window, spawned via the Task tool to handle a sub-task, returning only the useful result. Use them for codebase exploration, code review, parallel research, or any work that would bloat the main session. Define them in .claude/agents/*.md with their own system prompt, tool restrictions, and permission mode.
13. How do you handle rate limits and retries?
Implement exponential backoff with jitter on 429 and 5xx responses. Respect the retry-after header. Use the Batch API for non-real-time workloads — ~50% cheaper and on a separate quota. Spread load across model tiers. Cache stable prefixes to reduce token throughput. Monitor API usage per key to catch runaway loops early.
14. How do you defend against prompt injection?
Treat all untrusted input (user messages, web content, file contents, tool outputs) as data, not instructions. Wrap it in XML tags and explicitly tell Claude not to treat content inside as instructions. Constrain tools — never give destructive actions without human-in-the-loop confirmation. Use allow-lists for URLs and commands. Run evals with adversarial test cases. For high-stakes apps, add a classifier to flag suspicious inputs before they reach Claude.
15. How does extended thinking work and when should you enable it?
Extended thinking gives Claude a dedicated internal reasoning budget before producing its final answer. Set a thinking.budget_tokens parameter to control how long Claude reasons. Use it for tasks that benefit from deeper deliberation: hard math, complex logic, multi-step planning, or any situation where chain-of-thought alone isn't sufficient. Not recommended for simple tasks — it adds latency and cost.

CCA foundations — practice questions (click to reveal answers)

These are unofficial study questions modeled on official exam domains. Study the Anthropic Academy courses and docs.claude.com as your authoritative source.
Q1. Your customer-support agent handles 10K live conversations/day with a stable 5K-token system prompt. Which single feature most reduces your API cost?
Answer: Prompt caching. Mark the stable system prompt (and tool definitions) with a cache_control breakpoint. Cache hits dramatically reduce input token cost. The Batch API is cheaper still but only for async/offline workloads — live customer support requires real-time responses.
Q2. Which file does Claude Code automatically load at session start to provide project context?
Answer: CLAUDE.md. Claude Code reads CLAUDE.md from the project root (and ~/.claude/CLAUDE.md at user scope) at session start. It's the canonical location for conventions, architecture notes, and standing instructions.
Q3. What's the correct way to connect a remote MCP server directly via the Anthropic Messages API?
Answer: Use the MCP connector with the beta header anthropic-beta: mcp-client-2025-11-20 and pass the server URL in the mcp_servers parameter. This connects to remote MCP servers without needing a separate MCP client process running locally.
Q4. You're using Claude as a grader to evaluate another Claude's responses. What should you ask the grader to return for reliable scores?
Answer: Strengths, weaknesses, detailed reasoning, and a numerical score. Bare numbers produce near-random results. Requiring written justification forces the grader to engage with the content, producing consistent, defensible scores. This is the pattern Anthropic's evaluation guidance recommends.
Q5. Which Claude Code feature delegates a sub-task to an isolated context window that returns only its output?
Answer: Sub-agents. Defined in .claude/agents/*.md with their own system prompt, tool restrictions, and permission mode. Activated via the /agents command or the Task tool. They run with an independent context window and return only the useful result to the main session.
Q6. What's the most effective way to show Claude exactly what output format you want?
Answer: Few-shot examples. Provide one or more concrete output examples, ideally wrapped in tags. Concrete examples beat abstract format descriptions for adherence. Even a single high-quality example produces large gains in output consistency.
Q7. Which slash command compresses Claude Code conversation history while preserving a summary?
Answer: /compact. Unlike /clear (wipes all context), /compact summarizes prior turns and replaces them with the summary — typically reducing token usage 40–60% on long sessions while keeping the thread alive.
Q8. You need Claude Code to query a private Postgres database. What's the architecturally correct approach?
Answer: Build or install a Postgres MCP server that exposes database operations as MCP tools. Register it in settings.json at the appropriate scope (user, project, or local) with scoped credentials. This follows Anthropic's open standard and integrates cleanly with Claude Code's tool discovery.
Q9. Your agentic loop is making runaway tool calls. What's the first guardrail to add?
Answer: A maximum step / iteration cap. Pair it with a cost ceiling and timeout. Then add observability (log every tool call), evals on adversarial inputs, and human-in-the-loop confirmation for destructive or irreversible actions. Step caps are the cheapest, fastest-to-implement first line of defence.
Q10. Which model is best for high-volume, low-complexity classification at the lowest cost?
Answer: Claude Haiku 4.5. Fastest and cheapest current Claude model with near-frontier intelligence — designed for high-throughput classification, routing, simple extraction, and sub-agent roles where Sonnet or Opus would be overkill.
Q11. What does "tool use" mean in the Anthropic API and how does the loop work?
Answer: Tool use (function calling) lets you define JSON-schema-described tools. Claude returns a tool_use block when it wants to call one; your code executes the tool and feeds back a tool_result message. The model continues until it returns a final text answer with no further tool calls. Loop control (max turns, cost caps) is your responsibility.
Q12. What's the most important practice when designing tools for an agent?
Answer: Clear, descriptive tool names and descriptions with narrow scope per tool. Claude's tool selection accuracy depends heavily on tool metadata. Avoid mega-tools that do many things. Prefer focused tools with informative descriptions and explicit input schemas. Vague or overlapping tool descriptions are the leading cause of agentic failures.
Q13. How do you debug a silently failing MCP connection in Claude Code?
Answer: Launch Claude Code with --mcp-debug and run /mcp in-session to inspect server status. This surfaces handshake errors, authentication failures, and transport-level issues that aren't visible by default. Also check claude mcp list to verify configured servers.
Q14. Where should you place a coding convention you want shared with the whole team on every Claude Code session?
Answer: CLAUDE.md at the project root, committed to version control. Project-scope CLAUDE.md overrides user-scope and is shared via the repo — exactly the right pattern for team-wide conventions. User-scope (~/.claude/CLAUDE.md) is for personal preferences only.
Q15. The Batch API offers what tradeoff compared to the real-time Messages API?
Answer: ~50% lower cost in exchange for asynchronous, non-real-time completion. Submit a batch of requests; results return within a service window (up to 24 hours). Ideal for offline pipelines, evaluations, bulk data analysis, and labeling tasks where immediate responses aren't needed.
